# Phase 1: WMGP Ledger

## 目的

Katorin2 を `WMGP Season 8 を運営だけで最後まで回せる戦績管理台帳` として作り直す。

今回の主目的は、既存の Discord 中心運用を壊さずに、運営が Excel で持っていた情報を Web UI に置き換え、入力、修正、集計、画像出力の負担を減らすことにある。

## 前提

- ログイン必須
- 利用者は共通の運営アカウントを想定
- 参加者向け公開画面は作らない
- 参加者への共有は画像出力を基本とする
- チャット、掲示板、チーム向け自己管理 UI は Phase 1 で扱わない
- アプリケーション基盤は `Rails 8 + Hotwire` を第一候補とする
- DB は `Cloud SQL (PostgreSQL 17)` を使う

## スコープ内

- 複数リーグを管理できるアカウント構造
- `League` 配下でのチーム、参加者、画像出力管理
- `League > Phase` 配下でのブロック、Week、対戦管理
- 運営による手入力前提の台帳 UI
- Week 作成
- 対戦カードの手動作成
- 対戦日時の管理
- 試合結果の詳細入力
- 集計と順位計算
- 1対戦ごとの結果画像出力
- 決勝進出管理と決勝系台帳の運用

## スコープ外

- 単発大会
- 参加者向けポータル
- チームによるオーダー提出 UI
- チームによるデッキ提出 UI
- Discord の置き換え
- 公開向けマーケティング LP

## 現在の実装メモ

2026-03-20 時点の Rails 実装メモ:

- `League`, `Phase`, `Week`, `Match` は作成、編集、詳細表示まで入っている
- 業務データの削除導線はまだ Rails UI に移していない
- Rails 側で `DELETE` があるのは現状 `session` のログアウトだけ
- Active Record の関連には `dependent: :destroy` を入れているため、`League -> Phase -> Week -> Match` とその下位データは destroy 導線を追加すれば連鎖削除できる
- `Match` は物理削除より `cancelled` 運用の方が自然な場面が多いため、Phase 1 でどこまで物理削除を許可するかは別途決める
- 旧 Next.js 側には大会、参加者、招待、チームメンバーの削除が残っているが、Rails 移行後の正本機能とはまだ見なさない

削除まわりの次の判断ポイント:

- `League`, `Phase`, `Week`, `Match` のどこまでを `destroy` 対象にするか
- `Match` は `cancelled` を基本にし、物理削除は作成直後の誤登録だけに絞るか
- `Participant`, `Team`, `Block` を Phase 1 で Rails UI から編集、削除できるようにするか
- 削除時に監査ログや確認文言をどこまで要求するか

## 運用フロー

Phase 1 の基本フローは次で固定する。

1. Week を作成する
2. 対戦カードを手動で作成する
3. 対戦ごとに日時を設定する
4. 試合結果を入力する
5. 結果画像を作成する
6. 必要に応じて順位表と決勝進出を更新する

補足:

- `Phase` が予選と決勝を分ける
- `Week` は週番号のみを持つ
- 日付と時刻は `対戦` が持つ
- `Judge名`, `ルームID`, `観戦ID` は対戦作成時ではなく後追い入力を前提にする

## 情報設計

画面は `ホーム` と `台帳一覧` を分離する。

### ホーム

ホームは一覧ページではなく、運営のリアルタイム作業キューを出す。

優先表示候補:

- 今日の対戦
- ルーム情報未入力
- 結果未入力
- 画像未出力
- 要確認の試合

### リーグ配下

リーグを最上位オブジェクトとし、下位オブジェクトをすべて内包する。

- `League`
- `League > Phases`
- `League > Participants`
- `League > Teams`
- `League > Phase > Blocks`
- `League > Phase > Weeks`
- `League > Phase > Matches`
- `League > Exports`

`Teams` や `Participants` はトップレベル機能にしない。

## 認証

Phase 1 は運営専用アプリなので、認証は最小構成でよい。

方針:

- 運営だけがログインできる
- 共通運営アカウント運用を許容する
- 参加者やチーム向け権限は持たない
- 認証の複雑さより、運営の入りやすさを優先する

## 画像出力

最優先は `1対戦ごとの結果画像` である。

要件:

- WMGP の現行結果画像と実質同一の UI を出せること
- 正本デザインは `/home/iguchi/Downloads/WMGP/image1.png`
- デッキ名、並び順、W/L 表示、Judge、Week 表記、日付、総合スコアを含める

補足:

- 入力画面は結果画像と同じ見た目である必要はない
- 運営 UI はあくまで `Excel 台帳より速く入力できること` を優先する

# Rule Modules

## 方針

Phase 1 の UI と画像出力は WMGP 特化でよい。ただし、ルール、集計、報告照合まで WMGP 専用実装に固定すると、次リーグや Phase 2 で詰む。

そのため、内部は `コア` と `WMGP 固有実装` を分離する。

## 分離したい責務

### Core Domain

大会種別に依存しない台帳オブジェクトを扱う。

候補:

- League
- Week
- Match
- Match Result
- Round Result
- Board Result
- Export

### Core Rules

リーグや大会の進行ルールを差し替え可能にする。

候補:

- 総当たり
- 順位計算
- 決勝進出判定
- トーナメント進行

### Core Reports

結果入力と照合の責務を分離する。

候補:

- 公式記録
- チーム報告
- 差分判定
- 確定処理

### Core Renderers

同じ試合データを別レイアウトで描画できるようにする。

候補:

- `wmgpMatchResultRenderer`
- 将来の別リーグ用 renderer

## WMGP 固有で持つもの

WMGP 固有側で持つ責務:

- 予選ブロック運用の具体ルール
- 勝ち点とタイブレーク順序
- `3人メイン + 任意1人サブ`
- `最大3ラウンド / 2ラウンド先取`
- 結果画像の見た目
- 決勝ブラケットの見た目

これらは `core` に寄せず、`WMGP module` として閉じ込める。

## 現状コードとの関係

既存 Next.js 実装の `src/lib/tournament/*.ts` は、考え方の参照元としては使えるが、移植前提にはしない。

理由:

- 現行アプリは大会サービス前提の前史を強く引きずっている
- 今回は技術基盤ごと再構築する可能性が高い
- 先に責務境界を明文化し、新実装で必要な分だけ再実装した方が安全

したがって、既存コードは `参考資料` であり、`移植対象` ではない。

## 実装原則

- 入力 UI は WMGP に最適化してよい
- 集計ロジックは UI から独立させる
- 画像レイアウトは renderer として分離する
- 将来のチーム報告は `公式記録` と別レイヤーに置く
- まず WMGP を回し、その後に共通化する

## 初期ディレクトリ案

厳密な配置は実装時に調整するが、責務分離の基準は次を目安にする。

- `app/models`
- `app/services/rules`
- `app/services/reports`
- `app/renderers`
- `app/features/wmgp` または同等の名前空間

重要なのは名前よりも、`WMGP 固有ルール` と `使い回すコア` を混ぜないことである。

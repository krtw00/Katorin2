# Domain Model

## 階層

Katorin2 の基本階層は次で固定する。

`Account > League > Phase > Week > Match > Round`

`League` が最上位の業務オブジェクトであり、チームや参加者を横断共有しない。

## 基本方針

- 1つのアカウントで複数リーグを管理できる
- `Team` と `Participant` はリーグごとの閉じたスナップショットとして持つ
- 同名チーム、同一人物が別リーグに存在しても再利用しない
- Phase 1 は運営が `公式記録` を直接入力する
- 公式記録レイヤーと将来のチーム報告レイヤーを混ぜない構造を維持する (= 必要時に Plane Issue で再起票)

## 主要オブジェクト

### Account

- 運営アカウント
- 複数リーグの所有者

Phase 1 では共通アカウント 1 つでも運用可能だが、モデル上は複数リーグ管理を前提にする。

### League

リーグ単位で次の要素を内包する。

- 基本情報
- 運用ステータス
- Phases
- Blocks
- Teams
- Participants
- Weeks
- Matches
- Exports

`League` はシーズン単位の管理箱であり、WMGP Season 8 もこの単位で持つ。

### Phase

- リーグ配下の進行フェーズ
- `regular_season` や `playoff` を表現する
- ルールモジュールの切り替え単位
- 同じリーグの中に予選と決勝を共存させるための境界

`Phase` を挟むことで、予選と決勝を別プロダクトとして分けずに、同一リーグ配下で管理できる。

### Block

- リーグ配下の予選ブロック
- WMGP の現行運用をまず再現する
- 途中再編や可変ルールは Phase 1 では一般化しない

### Team

- リーグ配下のチーム
- 表示名
- 略称や画像出力用ラベル
- 所属ブロック
- 所属参加者

### Participant

- リーグ配下の参加者
- 所属チーム
- 表示名
- 必要に応じて画像出力に使う補助情報

### Week

- フェーズ配下の週オブジェクト
- `weekNumber` を持つ
- 日付は持たない

`Week` は台帳上の箱であり、実際の対戦日は `Match` に持たせる。

### Match

- 所属リーグ
- 所属 Phase
- 所属 Week
- 所属 Block
- 対戦チーム
- 対戦日
- 開始時刻
- Judge名
- ルームID
- 観戦ID
- 試合状態
- 画像出力状態
- 決勝用のブラケット位置など、Phase ごとの補助メタ情報

`Match` は運用上の最小管理単位であり、日時や進行管理の責務を持つ。

### Match Result

試合全体の公式結果。

含むもの:

- 総合スコア
- ラウンド勝敗
- 裁定情報
- 備考

### Round Result

1試合の中のラウンド結果。

含むもの:

- ラウンド番号
- 勝敗
- 時間切れや ED の裁定
- オーダー変更記録
- 各卓結果

WMGP では `最大3ラウンド / 2ラウンド先取` を前提にする。

### Board Result

各ラウンド内の 3 卓分の個人結果。

含むもの:

- 卓番号
- 自チーム側選手
- 相手チーム側選手
- デッキ名
- 勝敗
- マッチ得失点に必要な情報

結果画像はこの粒度まで必要とする。

### Export

- 結果画像の生成単位
- 対象 Match
- レイアウト種別
- 出力状態
- 生成日時

Phase 1 の主対象は `match-result image` である。

## 拡張ポイント

ドメインを直接拡張するのではなく、 `Core / WMGP module 分離` (= `rule-modules.md` 参照) で差し替え可能にする。

差し替えの対象になる責務:

- 順位計算ルール (= 勝ち点 / タイブレーク)
- 進行ルール (= 総当たり / スイス / ダブルエリミ等)
- 決勝進出判定
- 画像レンダラ

これら以外の Core Domain (= `League / Phase / Week / Match / Round` 等) はリーグ種別に依存しない形で維持する。

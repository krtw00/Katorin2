## 新規論点 (= 前回見落とし or s8 後半に発生)

### N-1. 警告 / 違反履歴の DB 化

- **生声**: 運営/keikoku ch 5 件 (= 4/10 集中、 `@team deck` `@team no game` 形式で告知のみ)。 履歴は Discord 上にフラットに並ぶだけで DB 未保存
- **影響**: 累積警告 (= 退場判断 / 過去傾向) が個人記憶依存。 シーズン跨ぐとリセット
- **推奨**: feature。 `warnings` テーブル (= team_id, week_id, category, judge, note) + チーム詳細から確認 UI

### N-2. 没収スコア自動反映 + DROP 種別記録

- **生声**: s8/drop ch 5 件 (= 5月中旬、 root judge が `@team DROP` 告知のみ)。 「強制 DROP」 3 件 / 「自主 DROP」 2 件で運営内では区別あり、 DB は未保存。 没収後の残試合スコア (= 1-2 / 得失点 -1) は **手作業で順位反映**
- **影響**: 没収スコア自動反映は見通しダッシュボードの残作業として認識済み。 加えて DROP 種別 (= 強制 / 自主) 区別の要件が判明
- **推奨**: feature。 KAT-28 の流れで `drop_kind` enum (forced / voluntary) を team or participant に追加 + 没収スコア自動展開

### N-3. R2/R3 結果入力の先読みデータ問題

- **生声**: 運営チャット 5/17。 R2/R3 のデータが結果入力 UI に先読みで埋まっていて未確定段階で表示 → 入力時に手作業削除が必要
- **影響**: 結果入力時の混乱 + UI 複雑化。 KAT-30 の R1 自動反映 (`7ba1bf7`) で R1 は解消したが R2/R3 は同じ問題が残る
- **推奨**: enhancement。 R2/R3 にも初期化ボタン or 未確定段階の grey out

### N-4. ルーム計画 / ジャッジプール (= 前回判断の再評価)

- **生声**: 運営/help-judge 126 msg。 4/10〜12 と 4/25 にラッシュ集中、 ルーム要請から提供まで 5〜40 分のばらつき、 深夜帯の遅延深刻、 ジャッジ習熟度差で「書き方わからない」 複数件
- **前回判断**: 「試合中に Katorin を見ない以上 Discord が最適」 → 未起票で確定 (`038b564`)
- **再評価**: **試合中の即応調整は Discord のままで正解**。 但し **事前計画 (= 週次ルーム数見積もり + ジャッジ TZ 別アサイン)** はアプリ支援可能 (例: week 一覧に「想定ルーム数 / 想定ジャッジ数 / TZ 別空き」)
- **推奨**: enhancement。 ただし KAT-25 (TZ 試合日程フロー) と一体設計が良い

## 既存 KAT-25〜31 補強情報 (= 定量化された新証拠)

### KAT-25 (TZ 試合日程フロー)
- s8/試合日程-battle-schedule **192 msg** / 投稿者 4 種で **運営 72%**
- 日付形式 3 種混在 (= 日本語 / 英文 / JST 表記)、 `Invalid date` 文字列 3 件、 AM/PM・12h/24h 取違いが week 3〜5 で再発
- → 自動 parse + UTC 内部管理 + フォーマット統一 UI が必須

### KAT-26 (参加者向け公開ページ)
- NOTICE/question-room-english **87 msg** の分類: ルール・制度 11 / 試合スケジュール 7 / **結果・順位 5** (= 「Group D の match レポが見れない」 直訴あり) / エントリー 3
- → 計 **26 件 (= 30%)** が公開ページで直接解消可
- NOTICE 系: TZ 情報非公開で日程調整 2 日遅延 / プレイオフ詳細非公開 / 順位表詳細非公開
- → en 切替は既存実装、 公開 + 国際化が同時に解ける

### KAT-27 (ロスター変更セルフ申請)
- s8/メンバー変更 **144 msg** の申請パターン: 追加 64% / 削除 14% / ID 追加 15% / ID 変更 5% / リーダー権限 2%
- **「@運営 staff」 催促** 16 件 (= 11%)、 申請後 3〜5 日の確認待ちが pain
- s8/チーム移動-team-to-team-transfer 5 件も同じ手転記パターン
- ID 形式が多様 (= MD ID 以外も混入)
- → 申請フォーム + 進捗可視化 + 形式バリデーション

### KAT-29 (結果カード Discord 自動投稿)
- s8/試合結果 **215 投稿が全件 text 空 + 画像のみ** (= 100% アプリ生成カードの手動再投稿)
- **3 アカウントで 78%** (= 運営 judge 58% / alliance 16% / cele 4%)、 week が進むほど投稿密度上昇 (= week1 56 → week5〜7 91)
- → webhook 自動投稿で 100% 人手削減可能、 **ROI 最高**

### KAT-30 / KAT-31 (結果入力フロー)
- 5月下旬で複数細部改善が進行 (= lineup R1 自動反映 / チーム管理導線 / DQ 選手名省略)
- 残るは **複数人同時編集** (= 並列 12 試合と相性最悪、 設計から見直し)
- スコアシート画像生成 (5/16 停止報告) は VPS 移行直後の bug、 直近 commit で復旧確認可

## 推奨優先順位 (= 次に手を付けるなら)

ROI と s8 残期間 (= プレイオフ進行中、 6月末ごろ終了想定) を踏まえると:

1. **KAT-29 (結果カード自動投稿)** — 100% 手作業 215 投稿、 ROI 最高、 webhook 1〜2 週で実装可
2. **KAT-26 (公開ページ)** — question-room-english の 30% 直接解消 + 国際参加者周知が Discord 手作業から脱却、 i18n 既存実装あり
3. **N-2 (没収スコア + DROP 種別)** — KAT-28 の流れで小規模実装可
4. **KAT-27 (ロスター変更セルフ申請)** — 144 件の手作業削減、 但しスキーマ + UI で工数あり
5. **KAT-25 (TZ 試合日程フロー)** — 紛争源、 但し UI 設計が重い (= s9 起こす前提なら s8 終了後に着手)
6. **N-1 (警告履歴 DB 化)** — シーズン跨ぐ価値、 短期 ROI は中
7. **N-3 (R2/R3 先読みデータ)** — KAT-30 の延長、 小修正
8. **N-4 (ルーム計画 / ジャッジプール)** — KAT-25 と一体設計推奨
9. **KAT-31 (複数人同時編集)** — 残るが設計重い

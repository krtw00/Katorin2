# Rebuild Architecture

## 概要

Katorin2 は既存の `公開寄り大会サービス` を削って延命するのではなく、`WMGP 運営専用台帳` として再構築する。

2026-03-19 時点の方針:

- アプリケーションは `Rails 8` を第一候補にする
- UI は `Hotwire (Turbo + Stimulus)` を基本にする
- DB は `Cloud SQL (PostgreSQL 17)` を使う
- デプロイ先は `GCP Cloud Run`

## なぜ全面作り直すか

現行リポジトリは、公開向け大会サービスとして積み上がった概念を多く抱えている。

観測できる負債:

- `supabase/migrations` が 32 本ある
- `src/app/[locale]/(main)` 配下に大会サービス前提の画面が多数残る
- `teams`, `tournaments`, `my page`, `i18n`, `公開導線` など、今回不要な概念が多い

今回必要なのは `運営専用の内部台帳` であり、既存構造を削って寄せるより、新しい土台を切った方が速く、安全で、判断もぶれにくい。

## 採用候補比較

### Rails 8

第一候補。

向いている理由:

- 複雑な関連を持つ業務データに強い
- CRUD とフォーム中心の内部画面を速く作れる
- 認証、バリデーション、添付、ジョブなどが標準に揃っている
- 将来の `報告 -> 照合 -> 確定` ワークフローを載せやすい
- Cloud Run で素直に運用できる

### Laravel

次点候補。

強い点は多いが、今回のチームが Ruby に抵抗がなければ Rails の方が業務アプリとして一段自然。

### Phoenix LiveView

技術的には適合度が高いが、導入学習コストが高い。今回は最短で WMGP を回すことが先なので、優先度は落とす。

### Next.js 継続

現行資産の流用余地はあるが、今回の主役は `フォーム`, `台帳`, `複雑な関連`, `内部向けワークフロー` なので、React 主導の構成で持ち続ける利点が薄い。

## 採用スタック

Phase 1 の推奨スタック:

- `Ruby 3.4+`
- `Rails 8.1`
- `PostgreSQL`
- `Hotwire (Turbo + Stimulus)`
- `Active Storage`
- `Solid Queue`
- `Cloud Run`

## DB

Cloud SQL (PostgreSQL 17) インスタンス `main-pg` を使用。
環境ごとに別データベース（`katorin2`, `katorin2-staging`）で分離。
Cloud Run からは Unix socket 経由で接続。

## 認証方針

Phase 1 は `運営専用のシンプルなログイン` を優先する。

推奨:

- Rails 側で認証を完結させる
- アカウントは少数の運営用に限定する
- 参加者やチーム用の複雑な認可は持たない

理由:

- 今回は共通運営アカウント前提
- 参加者向けログインは不要
- 将来のチーム報告導入まで、細かい権限管理は過剰

## アプリ構造

責務は次の単位で切る。

- `app/models`
  - League, Block, Team, Participant, Week, Match, Export
- `app/controllers`
  - Organizers, Leagues, Matches, Exports
- `app/services`
  - 集計、順位計算、決勝進出、画像データ整形
- `app/renderers`
  - WMGP 結果画像レイアウト
- `app/jobs`
  - 画像生成、再集計
- `app/policies` または同等の認可層
  - 運営専用アクセス制御

重要なのは、`WMGP 固有レイアウト` と `再利用可能な業務ロジック` を分けること。

## 画像出力

画像出力はサブ機能ではなく、Phase 1 の中心機能である。

方針:

- 1対戦結果画像を最優先
- `image1.png` を正本レイアウトとして再現
- 画像生成は同期でもよいが、詰まるなら `job 化` できるようにする
- 将来の順位表画像、決勝画像は renderer を分けて足す

## 移行方針

全面移行はせず、`別アプリとして切り直す` 前提で進める。

推奨手順:

1. 既存 Next.js 実装は参照用として保持する
2. Rails アプリを別ディレクトリまたは別ブランチで新規作成する
3. WMGP Phase 1 の最小スキーマから migration を切る
4. 運営ログイン、League、Week、Match 入力を先に作る
5. 集計と画像出力を載せる
6. 最後にホームの作業キューを整える

## 破棄する前提

再構築時に引きずらないもの:

- 公開向け大会一覧
- 単発大会中心の情報設計
- 参加者向けマイページ
- チーム公開導線
- i18n 前提
- Realtime 前提

## 引き継ぐもの

再利用候補:

- WMGP の業務知識
- 既存の対戦進行ロジックの考え方
- 画像 API の出力要件
- GCP デプロイ手順
- 過去 UI から得た運営導線の知見
